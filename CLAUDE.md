## Serena + JetBrains Is the Default (CRITICAL)

**MANDATORY**: Serena (JetBrains IDEA) is the **primary tool for ALL code work**. Claude Code `Read`/`Grep`/`Glob`/`Edit` are **only** for non-code files (`.md`, `.yml`, `.json`) or files outside this project.

| Task | Serena Tool |
|---|---|
| File structure | `jet_brains_get_symbols_overview(file, depth=1)` |
| Find symbol | `jet_brains_find_symbol(name)` |
| Read source | `jet_brains_find_symbol(name, include_body=True)` |
| Find callers | `jet_brains_find_referencing_symbols(name, file)` |
| Type hierarchy | `jet_brains_type_hierarchy(name, file)` |
| Library APIs | `jet_brains_find_symbol(name, search_deps=True)` |
| Regex search | `search_for_pattern(pattern)` |
| Find files | `find_file(mask, path)` |
| Replace symbol | `replace_symbol_body(name, file, body)` |
| Insert code | `insert_before/after_symbol(name, file, content)` |
| Rename | `rename_symbol(name, file, new_name)` |

**Workflow**: Overview first → drill down → trace usage → check hierarchy → edit.

**Key rules**:
- ALWAYS `get_symbols_overview` before reading any code file
- ALWAYS `find_referencing_symbols` before renaming or changing signatures
- ALWAYS `type_hierarchy` before modifying interfaces
- Pass `relative_path` whenever possible for faster results
- `find_referencing_symbols` requires a file path, not directory
- Claude Code `Edit` is OK for surgical edits (3-5 lines) inside large functions

**Exceptions for Bash/Grep**: Counting things (`grep -c`, `wc -l`), route patterns in `main.go`, text in comments/strings.

**NEVER delegate code edits to sub-agents (Task tool) with `isolation: "worktree"`**. Worktree agents cannot use Serena because JetBrains IDE only indexes the main working directory. ALL code reading and editing MUST happen in the main agent using Serena tools. Sub-agents may ONLY be used for non-code tasks (running tests, searching docs, web fetches). If a task requires editing many files, the main agent does it directly with Serena — no shortcuts.

**Serena memories**: `project_overview`, `suggested_commands`, `architecture/go_backend`, `architecture/angular_web`, `conventions/go_style`, `conventions/angular_style`, `task_completion_checklist`, `gotchas`

## Learned Patterns

### PayOS
- PayOS sends test webhooks on URL verification — handler must return `200 OK` before signature check for test payloads (`"VQRIO123"`)
- Client-controlled `return_url`: web sends `window.location.origin + "/payment/return"`, Android falls back to server handler
- `.env` is NOT loaded when `GO_ENV=production` — use system env vars

### Dual Component Architecture (Web)
- Production uses `app-task-card` (`TaskCardComponent`), NOT `nhannht-metro-task-card`
- `nhannht-metro-*` are design system references (Storybook only). `shared/` is production.
- Always trace imports from page components before modifying shared components

### Storybook
- All `@storybook/*` packages must be on the same major version (mixing causes CJS/ESM errors)
- `@angular/platform-browser-dynamic` is REQUIRED (without it, infinite loading spinner)
- Compodoc JSDoc must go **before** `@Component()` decorator

## Claudtest (ADB Device Testing)

Automated testing via ADB. Principles: **batch commands** (chain in one `adb shell` call), **minimize screenshots** (only at screen transitions), **visual estimation first** (fallback to `uiautomator dump` only if estimation fails twice).

```bash
# Batch example — ONE adb call instead of 4
adb shell "input tap 380 690; sleep 0.3; input text 'test@example.com'; input tap 380 820; sleep 0.3; input text 'password123'; input tap 380 950"
```

Use for: verifying UI fixes, reproducing bugs, testing user flows. Not a replacement for unit tests.
